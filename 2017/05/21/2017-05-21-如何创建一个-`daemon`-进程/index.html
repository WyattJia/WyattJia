如何创建一个 `daemon` 进程

Author: Wyatt Jia

daemon 进程相对于 普通 的进程，其具备下列特征：

  • 生命周期长。通常, 一个 daemon 进程会在系统启动时被创建，一直运行到系统关闭时被被关闭。
  • 运行在后台，并且几乎不控制终端。缺失控制终端之后，确保了内核永远不会为 daemon 进程自动生成任何任务控制信号以及终端相关的信号。

要想将一个普通进程变成 daemon 进程，一个程序首先要完成下面的步骤：

  • 执行一个 fork() ，之后父进程退出，子进程继续执行。（其结果是 daemon 进程变成了 init 进程的子进程）。之所以这么做是因为下面两个原因。
    • 假设 daemon 进程是由命令行启动的，父进程的终止会被 shell 发现，shell 在发现之后显示出另一个 shell 提示符并让子进程继续在后台运行。
    • 子进程被确保不会成为一个进程组首进程，因为它从其父进程那里继承了进程组 ID 并且拥有了自己的唯一的进程 ID ，这个进程 ID 跟 继承而来的进程组 ID 是不同的。这样才能确保继续下一步。
  • 子进程调用 setid() 开启一个新会话并释放它与控制终端之间的所有关联关系。
  • 如果 daemon 从来没有打开过终端设备，那么就无需担心 daemon 会重新请求一个控制终端了。如果 daemon 后面可能会打开一个终端设备，那么必须要采取措施来确保这个设备不会成为控制终端。这可以通过下面两种方式实现。
    • 在所有可能应用到一个终端设备上的 open() 调用中指定 O_NOCTTY 标记。
    • 或者更简单地说，在 setid() 调用之后执行第二个 fork() 然后再次让父进程退出并让孙子进程继续执行。
  • 清楚进程的 umask 以确保当 daemon 创建文件和目录时拥有所需的权限。
  • 修改进程的当前工作目录，通常为根目录 / 。
  • 关闭 daemon 从其父进程继承而来的所有打开着的文件描述符。
  • 关闭描述符之后，daemon 通常会打开 /dev/null 并使用 dup2() 使所有的这些描述符指向这个设备。这样做的原因如下：
    • 确保了当 daemon 调用了在这些描述符上执行 I/O 的库函数时不会出乎意料地失败。
    • 防止 daemon 使用描述符 1 或者 2 打开一个文件的情况。
